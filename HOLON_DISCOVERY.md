# Holon Discovery Spec

Status: draft, 
⚠️Partially implemented this document is the source of truth but the code may not be fully aligned.⚠️

## Installation Model

All holons are installed as `.holon` [packages](HOLON_PACKAGE.md) in `$OPBIN`:

```
$OPBIN/
  grace-op.holon/
    .holon.json
    bin/darwin_arm64/op
  rob-go.holon/
    .holon.json
    bin/darwin_arm64/rob-go
```


---

## Discovery Algorithm

This algorithm is a **cross-SDK spec**. 
Every SDK implements the same logic.
c, c++, c#, dart, go, java, js, js-web, kotlin, python, ruby, rust, swift. 


### Scan

1. Walk the root directory recursively
2. Match directories ending in `.holon`
3. For each `.holon` directory:
- **Fast path**: read `.holon.json` if present
- **Fallback**: launch the binary via stdio, call `HolonMeta.Describe`
4. Walk for `holon.proto` files (source holons)

### Search order

When resolving across all layers:

```
1. Bundle          Contents/Resources/Holons/*.holon/
2. Build output    .op/build/*.holon/
3. Installed       $OPBIN/*.holon/
4. Cache           $OPPATH/cache/**/*.holon/
5. Source          recursive holon.proto walk from root
```

First match by UUID wins. Each layer is scanned for both `.holon`
packages and source `holon.proto` files.

### Exclusion rules

Source tree walking skips:

- `.git`, `.op`, `node_modules`, `vendor`, `build`, `testdata`
- Any directory starting with `.` (except `*.holon` package directories)

### `.holon.json` as accelerator

- **Generated by `op`** from the proto at build time, never hand-edited
- **Not a hard requirement** — if missing, the SDK probes via stdio
  `Describe` to obtain metadata
- **Only `op` writes** `.holon.json` — SDKs read or probe, they don't
  cache

## `op list`

`op list` is the single command for listing holons. `op discover` is
removed.

### Flags

| Flag | Scope |
|------|-------|
| *(none)* | same as `--all` |
| `--all` | everything across all layers |
| `--source` | source holons in workspace |
| `--built` | `.op/build/` packages |
| `--installed` | `$OPBIN` packages |
| `--cached` | `$OPPATH/cache` packages |
| `--root <path>` | override CWD as scan root |
| `--format json` | JSON output |

### Default output columns

```
SLUG                    VERSION  NAME              LANG  STATUS  ORIGIN     NATURE   PATH
rob-go                  0.1.0    Rob Go            go    draft   installed  binary   rob-go.holon
grace-op                0.5.6    Grace Op          go    draft   installed  binary   grace-op.holon
gabriel-greeting-go     0.2.1    Gabriel Greeting  go    draft   source     source   holons/gabriel-greeting-go
my-transcriber-py       0.1.0    My Transcriber    py    draft   cached     dist     my-transcriber-py.holon
```

### Fields

| Column | Description |
|--------|-------------|
| SLUG | Canonical holon identifier (`given-family`, lowercased) |
| VERSION | Semver from identity (e.g. `0.5.6`) |
| NAME | Human name (`Given Family`) |
| LANG | Implementation language |
| STATUS | Lifecycle stage: `draft`, `stable`, `deprecated`, `dead` |
| ORIGIN | Where found: `source`, `built`, `installed`, `cached`, `bundle` |
| NATURE | Distribution form: `binary`, `dist`, `source` |
| PATH | Relative path to the holon |

ORIGIN, NATURE, and PATH are runtime-derived. The remaining fields come
from `.holon.json`. Fields to **add** to `.holon.json` and the proto:

- `version` — semver from the identity manifest
- `aliases` — list of alternative names (e.g. `["op"]` for `grace-op`)

`--format json` outputs all available fields from `.holon.json` per entry.

## Resolution Overrides

Three mechanisms bypass or constrain the default discovery[^1].

### `--root` — scoped discovery

Override the effective root so `op` scans a different directory tree:

```bash
# Call a holon discovered under ~/Desktop/isolated, not cwd:
op --root ~/Desktop/isolated gabriel-greeting-go SayHello '{"name":"","lang_code":"fr"}'
# → {"greeting":"Bonjour Marie","language":"French","langCode":"fr"}

# List holons under a specific directory:
op --root ~/Desktop/isolated discover
```

Sets `OPROOT` env var for the duration of the command.
All downstream discovery (`ResolveBinary`, `DiscoverHolons`) reads
`OPROOT` via `env.Root()`.

### `--bin` — print resolved binary path

Print the absolute binary path for a slug and exit (no RPC call):

```bash
op --bin gabriel-greeting-go
# → /Users/Bob/.op/bin/grace-op.holon/bin/darwin_arm64/gabriel-greeting-go

# Combine with --root for scoped resolution:
op --root ~/Desktop/isolated --bin gabriel-greeting-go
```

### Direct binary dispatch

When the first argument is an **existing executable file**, `op`
launches it via `stdio://` with no discovery[^2]:

```bash
op ./.op/build/grace-op.holon/bin/darwin_arm64/gabriel-greeting-go \
   SayHello '{"name":"World"}'
```

Detection heuristic: the argument must contain `/`, `\`, or `.`,
point to an existing non-directory file, and have the executable
bit set. If the file does not exist, `op` falls through to slug
resolution.

[^1]: All three are global flags parsed before command dispatch.
[^2]: The binary is launched with `serve --listen stdio://`, same as `grpc+stdio://`.
